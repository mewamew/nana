# 记忆系统调研报告

## 调研背景

当前 Nana 项目的记忆系统问题：
- 归档策略粗暴（超过 20 条直接把前 10 条原始文本存入 ChromaDB）
- 原始对话文本语义密度低，检索质量差
- 用户档案（me.txt）更新不可靠

## 调研选项

### 方案 A：LLM 摘要归档（已选定）

**原理**：归档前调用 LLM 对旧对话生成摘要，存入向量库

**优点**：
- 摘要比原始文本语义更集中，检索质量更高
- 无额外依赖，使用现有 LLM provider
- 降级方案简单（LLM 失败时存原始文本）
- 对现有代码改动最小

**缺点**：
- 归档时额外消耗一次 LLM 调用
- 摘要质量依赖所选模型

**结论：采用此方案**（见 `05-memory-upgrade.md`）

---

### 方案 B：Mem0 框架

**原理**：专门的记忆管理框架，自动从对话中提取结构化记忆

**优点**：
- 自动提取实体、事件、偏好等结构化信息
- 支持增量更新（更新已有记忆而非追加）

**缺点**：
- 额外依赖（`mem0ai` 包）
- 有 API 调用费用（使用托管服务时）
- 与现有 ChromaDB 存储方式不兼容，需要迁移

**结论：暂不采用，留作未来升级方向**

---

### 方案 C：分层记忆架构

**原理**：
```
工作记忆（最近 N 轮，RAM）
情节记忆（摘要，ChromaDB）
语义记忆（用户偏好/事实，结构化 JSON）
```

**优点**：最完整的记忆架构

**缺点**：
- 实现复杂度较高
- 需要较大改动现有架构

**结论：方案 A 实际上已包含了分层记忆的核心思想，未来可在 A 基础上逐步演进**

---

## 最终决策

采用**方案 A（LLM 摘要归档）**，理由：
1. 改动最小，风险最低
2. 效果提升明显（摘要 vs 原始文本）
3. 完全兼容现有 ChromaDB 存储
4. 降级机制完善

具体实现见 `specs/05-memory-upgrade.md`。
